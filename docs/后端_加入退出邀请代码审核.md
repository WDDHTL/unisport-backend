# 后端加入/退出邀请接口代码审核

- 评分：7.5/10
- 范围：`InviteController` 的加入/退出接口（`src/main/java/com/unisport/controller/InviteController.java:99-118`），以及对应的服务实现 `InviteServiceImpl`（`src/main/java/com/unisport/service/impl/InviteServiceImpl.java:197-308`、`471-520`）。
- 结论：基础鉴权、校区隔离、事务与并发控制思路较完整，但在锁降级策略、数据约束与缓存清理上存在风险。

## 优点
- 登录校验与学校隔离贯穿全链路，避免跨校访问。
- 关键写操作包裹在事务中，加入/退出均校验状态并使用 `UpdateWrapper` 保护名额。
- 通过 Redis 锁和 DB 条件更新的组合减少并发超卖风险。

## 问题与风险
1. **加入/退出锁失败时直接放行，缺少 DB 层唯一保护**（高）  
   - 位置：`acquireJoinLock` 在 Redis 异常时返回 `true`（`InviteServiceImpl.java:511-517`），`joinInvite` 在获取不到锁时才拒绝（`InviteServiceImpl.java:209-257`）。`invite_members` 未见唯一约束，插入前的 `findMember` 仅依赖应用层判断（`InviteServiceImpl.java:218-245`）。  
   - 影响：Redis 故障或网络抖动时锁会失效，多个并发请求可能同时插入同一用户的多条成员记录并多次累加 `joined_count`，导致名额被虚增、用户重复加入，破坏数据一致性。  
   - 建议：锁失败时采取“失败关闭”策略（直接报错或退化为数据库乐观锁）；同时在表层添加 `(invite_id, user_id)` 唯一索引，并在插入时处理唯一冲突。

2. **请求字段被静默忽略**（中）  
   - 位置：`JoinInviteDTO` 提供 `comment`、`fromShareToken`（`src/main/java/com/unisport/dto/JoinInviteDTO.java`），但 `joinInvite` 全流程未使用或持久化这些字段（`InviteServiceImpl.java:197-257`）。  
   - 影响：客户端提交的备注、来源信息被直接丢弃，业务需求（记录来源、风控或通知个性化）无法满足，且前端难以感知。  
   - 建议：明确需求，若需要存储则扩展 `invite_members`/日志表并写入；若暂不支持，应在接口契约中移除或标记为未使用，避免误导。

3. **缓存清理使用 `KEYS` 全量扫描**（中）  
   - 位置：`evictInviteCaches` 通过 `keys("invite:*")` 扫描并删除（`InviteServiceImpl.java:794-807`）。  
   - 影响：在生产 Redis 上，`KEYS` 属于阻塞式全量扫描，键数量大时会造成 Redis 主线程阻塞，影响所有请求；且同一键空间下其他业务也可能被误删。  
   - 建议：改用按前缀的 Set/索引记录或 `SCAN` 游标渐进删除，或为列表缓存加版本号前缀、只淘汰当前版本，避免全量 `KEYS`。

4. **锁粒度单一且 TTL 固定，易出现长事务穿透**（低）  
   - 位置：加入/退出共用 invite 级锁 `lock:invite:join:{id}`，TTL 10s（`InviteServiceImpl.java:207-213`, `272-275`, `511-517`）。  
   - 影响：当单次事务异常拉长（慢 SQL/网络），锁超时后并发请求可进入同一流程，可能与正在执行的事务产生写写冲突（尽管 DB 条件更新能拦截部分，但成员状态更新仍可能出现回滚/覆盖）。  
   - 建议：在锁值中写入唯一 token，释放时校验 token；必要时延长 TTL 或使用续期机制，避免长事务期间锁失效。

## 结论与建议
- 当前逻辑在正常情况下可避免简单的超卖，但对基础设施故障（Redis 不可用）和高并发长尾场景的防护不足，建议优先修复高风险项：失败关闭锁策略 + 数据库唯一约束。  
- 同步明确 `JoinInviteDTO` 字段使用方式，避免接口契约与实现不一致。  
- 优化缓存淘汰策略，替换 `KEYS` 以降低对 Redis 的潜在阻塞风险。
