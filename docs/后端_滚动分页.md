# 后端_滚动分页

## 背景
- 现状：`GET /posts?categoryId=` 直接返回所在学校、指定分类的全部帖子列表，`PostServiceImpl#getPostList` 使用 `selectList` 无分页，`PostQueryDTO.current/size` 未被使用。
- 诉求：改造成滚动分页（cursor based），避免大列表一次性下发，提升移动端滚动体验和服务端性能。
- 既有能力：数据库已有 `idx_category_time (category_id, created_at desc)` / `idx_created` 索引，通用分页封装 `PageResult` 已存在，可复用/参考。

## 复杂度评估
- **改动范围可控**：仅涉及 `PostQueryDTO`、`PostController`、`PostService` 接口与 `PostServiceImpl` 查询逻辑，Mapper 与数据库结构无需改动。
- **实现难度中等偏低**：MyBatis-Plus 支持 `limit`/`orderBy` 组合，按 `(created_at, id)` 组合游标实现去重容易落地。
- **兼容性注意点**：响应体由原来的 `List<PostVO>` 变为分页对象，需要前端同步适配；接口文档需更新。
- **性能风险低**：索引已覆盖查询条件；查询只取 `size+1` 条用于判断 `hasMore`，单次请求开销与列表大小无关。

## 设计与改造方案
### 请求/响应设计
- 请求参数：
  - `categoryId`：必填。
  - `cursor_time`（可选，ISO8601 字符串）；`cursor_id`（可选，Long）。
  - `size`：可选，默认 10，最大建议 50。
- 响应体：推荐新增轻量封装 `ScrollPageResult<T>`：`records`、`hasMore`、`nextCursorId`、`nextCursorTime`、`size`。如需保留总数可继续用 `PageResult`，但滚动场景通常不返回 `total/pages`。

### 代码改动清单
1) **DTO** `PostQueryDTO`：新增 `LocalDateTime cursorTime`、`Long cursorId`，保留 `size` 并设默认值，移除未使用的 `current`。  
2) **分页响应类**：新增 `common/ScrollPageResult.java`（若不想动 `PageResult`）。
3) **Controller** `PostController#listPosts`：
   - 参数增加 `cursorTime`、`cursorId`、`size`（含默认值）。
   - 返回值改为 `Result<ScrollPageResult<PostVO>>`（或 `PageResult` 版本）。
4) **Service 接口** `PostService`：`getPostList` 返回类型同步调整，参数使用新的 DTO 字段。
5) **Service 实现** `PostServiceImpl#getPostList`：
   - 校验用户 & 学校逻辑保持。
   - 查询条件：`eq schoolId/categoryId`；若传入游标，按 `(created_at, id)` 组合做 `<` 过滤，处理同一时间戳时使用 `id` 打破并发写入的排序平局。
   - 排序：`orderByDesc(createdAt).orderByDesc(id)`。
   - 分页：`limit size+1` 获取是否还有更多，超过时弹出多余一条并设置 `hasMore=true`。
   - 封装 `nextCursorTime/id` 为当前返回列表最后一条的 `(createdAt, id)`。
   - 保留现有 `PostVO` 填充用户昵称/头像、`isLiked` 默认逻辑。
6) **文档**：更新 API 文档及 OpenAPI 注解，补充请求示例和响应示例。

### 关键查询伪代码
```java
LambdaQueryWrapper<Post> qw = new LambdaQueryWrapper<Post>()
    .eq(Post::getSchoolId, schoolId)
    .eq(Post::getCategoryId, dto.getCategoryId());

if (dto.getCursorTime() != null) {
    qw.and(w -> w
        .lt(Post::getCreatedAt, dto.getCursorTime())
        .or()
        .nested(n -> n.eq(Post::getCreatedAt, dto.getCursorTime())
                      .lt(Post::getId, dto.getCursorId()))
    );
}
qw.orderByDesc(Post::getCreatedAt).orderByDesc(Post::getId);
List<Post> rows = postMapper.selectList(qw.last("limit " + (dto.getSize() + 1)));
```

### 建议的测试用例
- 不带游标首屏：返回最新 `size` 条，`hasMore` 根据总量判断。
- 带游标加载更多：确保新列表首条为上次最后一条之后的记录，无重复/漏数。
- 同一秒插入多条：验证 `(created_at, id)` 组合游标无跳漏。
- size 边界：`size=1`、超过上限时回落到上限。
- 无更多数据：`hasMore=false`，`nextCursor*` 为空。
- 权限/校验：缺失 `categoryId`、用户未登录、schoolId 为空的异常路径。

## 结论
- 改造为滚动分页的复杂度可控、无需数据库结构变更；主要工作量在接口签名与 Service 查询逻辑调整。
- 按上述方案改造后，可显著降低列表接口单次负载，提升移动端滚动体验。前后端需同步更新请求参数与响应解析。
