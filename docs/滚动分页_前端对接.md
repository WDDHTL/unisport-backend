# 滚动分页_前端对接

## 目标
- 与后端对齐帖子列表改为滚动分页（cursor-based）的请求参数、响应体和交互约定。
- 重点：避免重复数据、保证滑动体验流畅、兼容首屏/加载更多/刷新。

## 接口改动概览
- 路径：`GET /posts`
- 入参：新增游标 + size
  - `categoryId` (必填)：运动分类。
  - `cursor_time` (可选)：上一页最后一条的 `created_at`，ISO8601 字符串。
  - `cursor_id` (可选)：上一页最后一条的 `id`，与 `cursor_time` 搭配消除同秒并发写入导致的重复/跳漏。
  - `size` (可选)：默认 10，最大建议 50，前端可按屏幕适配调整。
- 出参：滚动分页结果（建议由后端返回 `ScrollPageResult<PostVO>`）
  - `records`：当前返回的帖子列表。
  - `hasMore`：是否还有更多。
  - `nextCursorTime` / `nextCursorId`：下一次请求要携带的游标；无更多时可为空。
  - `size`：本次请求使用的 size。

> 说明：后端查询顺序 `created_at desc, id desc`，游标过滤 `(created_at, id)` 双字段，避免并发插入导致的重复或遗漏。

## 约定补充（需对齐）
- 参数/字段命名：请求用下划线 `cursor_time` / `cursor_id`；响应字段 `nextCursorTime` / `nextCursorId`；点赞标记统一 `liked`（布尔），前端展示用本地命名转换。
- 时间格式：游标时间使用 ISO8601 字符串（示例 `2026-03-13T09:59:50Z`，含时区），后端解析为 UTC；前端传参保持 UTC，展示时按本地时区渲染。
- size 规则：默认 10，最大 50；超出上限后端回落到 50，不报错。
- 无更多时：`hasMore=false`，`nextCursorTime/Id` 为空或不存在；前端以 `hasMore` 为准停止加载。
- 兼容策略：若前端暂未升级，仅传 `categoryId` 将返回首屏（不带 cursor）；后续“加载更多”必须按新游标协议，否则会重复数据。
- 错误码：未登录/鉴权失败使用现有 401/403；缺少 `categoryId` 返回 400；size 非法同样 400；其余遵循通用 `code`/`msg` 结构，前端兜底提示“加载失败，点击重试”。
- 文档/类型：后端会更新 OpenAPI 注解；前端 TS 类型/Mock 按此字段落地；如需临时兼容开关请注明。

## 请求/响应示例
```http
GET /posts?categoryId=1&size=10
```
```json
{
  "code": 0,
  "data": {
    "records": [
      {
        "id": 123,
        "userId": 9,
        "userName": "Jason",
        "userAvatar": "...",
        "categoryId": 1,
        "content": "hello",
        "images": null,
        "likesCount": 3,
        "commentsCount": 1,
        "liked": false,
        "createdAt": "2026-03-13T10:00:00"
      }
    ],
    "hasMore": true,
    "nextCursorTime": "2026-03-13T09:59:50",
    "nextCursorId": 122,
    "size": 10
  }
}
```

加载更多请求：
```http
GET /posts?categoryId=1&size=10&cursor_time=2026-03-13T09:59:50&cursor_id=122
```

## 前端对接要点
- **首屏**：`cursor_time`/`cursor_id` 为空即取最新；渲染后缓存末条的 `(createdAt, id)`。
- **上拉加载更多**：使用上次响应的 `nextCursorTime` + `nextCursorId` 作为游标；`hasMore=false` 时停止请求并展示“没有更多了”。
- **去重策略**（防重复渲染）：
  - 使用帖子 `id` 作为唯一键；新 records 追加前做 `Set`/Map 判重。
  - 若本地已有同 `id`，以时间靠后的数据覆盖；后端排序保证新增记录不会逆序插入，但并发刷新时覆盖更安全。
- **并发与节流**：滚动加载时需锁定请求状态，避免同一游标发起多次请求造成重复；可用 `loading` flag 或请求队列。
- **下拉刷新**：刷新应清空列表与游标，从头拉取最新数据；避免继续使用旧 `cursor`。
- **分页大小**：移动端建议 10~20，Pad/桌面可适当放大；超出后端上限会被后端回落。
- **异常/空态**：网络错误保持当前列表，允许重试；空列表时展示空态组件；`hasMore=false` 且 records 为空时提示“暂无内容”。
- **埋点**：如有曝光/点击埋点，注意列表追加后重新计算可见区域，防止重复曝光。

## 前后端联调清单
- [ ] 首屏返回正常，cursor 字段齐全。
- [ ] 同一游标重复请求不会重复渲染（前端去重生效）。
- [ ] 同一秒多条数据无跳漏（验证 `(created_at,id)` 游标）。
- [ ] size 边界（1、超上限）按预期回落。
- [ ] hasMore=false 时正常停止加载更多。
- [ ] 下拉刷新重置状态后能看到最新数据。
