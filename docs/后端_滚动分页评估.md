# 后端_滚动分页评估

## 1. 概述

本文档评估 UniSport 后端帖子列表接口实现滚动分页（Cursor-based Pagination）的可行性，并给出具体的改动建议。

**当前状态**：帖子列表接口 `GET /api/posts` 一次性返回所有帖子数据，不支持分页。

**目标**：支持滚动分页，提升大数据量下的用户体验和系统性能。

## 3. 滚动分页方案评估

### 3.1 两种分页方式对比

| 特性 | 传统分页 (OFFSET/LIMIT) | 滚动分页 (Cursor-based) |
|------|-------------------------|------------------------|
| **原理** | 使用 LIMIT offset, size | 使用 WHERE cursor < last_created_at |
| **适合场景** | 固定页码跳转（如1,2,3页） | 无限滚动（Feed流） |
| **性能** | 大偏移量时性能下降严重 | 性能稳定，不受数据量影响 |
| **数据一致性** | 翻页时数据可能变化 | 适合实时数据流 |
| **实现复杂度** | 简单 | 略复杂 |
| **用户体验** | 支持跳页 | 只能向前/向后滚动 |

### 3.2 滚动分页的优势

1. **性能稳定**：无论翻到第几页，查询时间基本一致（O(1) 而非 O(n)）
2. **适合移动端**：社交类 App 主流采用滚动分页
3. **避免重复/遗漏**：数据新增时，滚动分页不会导致数据重复显示

### 3.3 滚动分页的挑战

1. **不支持跳页**：用户无法直接跳到第5页
2. **边界处理**：需要处理没有更多数据的场景
3. **双向滚动**：如果需要支持"上拉刷新加载历史"，需要额外处理

## 4. 可行性结论

### ✅ **结论：完全可行**

理由：
1. 数据库已建立合适的索引 `idx_category_time` 和 `idx_created`
2. 项目已有分页基础设施可复用
3. 改动范围可控，仅涉及 Controller 和 Service 层

### 📋 **推荐方案：游标分页 (Cursor Pagination)**

使用 `created_at` + `id` 作为游标，兼容以下场景：
- 首次加载：传入 `null` 游标，返回最新数据
- 下拉加载更多：传入上一页最后一条记录的 `created_at` 和 `id`
- 上拉刷新：传入最新数据的游标

### 5.4 Service 实现改动

**文件**: [service/impl/PostServiceImpl.java](file:///e:\unisport-backend\src\main\java\com\unisport\service\impl\PostServiceImpl.java#L112-L157)

```java
@Override
public PageResult<PostVO> getPostList(PostQueryDTO postQueryDTO) {
    // 获取当前用户的学校ID
    Long userId = UserContext.getUserId();
    Long schoolId = getSchoolId(userId);
    
    if (schoolId == null) {
        throw new BusinessException("学校信息缺失");
    }

    // 构建查询条件
    LambdaQueryWrapper<Post> queryWrapper = new LambdaQueryWrapper<>();
    queryWrapper.eq(Post::getSchoolId, schoolId)
                .eq(Post::getCategoryId, postQueryDTO.getCategoryId());
    
    // 游标分页核心逻辑
    if (postQueryDTO.getCursorTime() != null) {
        // 使用 (created_at, id) 组合游标
        queryWrapper.lt(
            Post::getCreatedAt, 
            postQueryDTO.getCursorTime()
        );
        // 如果同时间戳，再按ID排除
        if (postQueryDTO.getCursorId() != null) {
            queryWrapper.and(w -> w
                .lt(Post::getCreatedAt, postQueryDTO.getCursorTime())
                .or()
                .eq(Post::getCreatedAt, postQueryDTO.getCursorTime())
                .lt(Post::getId, postQueryDTO.getCursorId())
            );
        }
    }
    
    // 按时间倒序、ID倒序排序
    queryWrapper.orderByDesc(Post::getCreatedAt)
                .orderByDesc(Post::getId);
    
    // 分页查询
    Page<Post> page = postMapper.selectPage(
        new Page<>(1, postQueryDTO.getSize()), 
        queryWrapper
    );
    
    // 转换为VO
    List<PostVO> voList = convertToVO(page.getRecords());
    
    // 返回分页结果（携带游标信息供前端使用）
    return PageResult.of(
        1, 
        postQueryDTO.getSize(), 
        page.getTotal(), 
        page.getPages(), 
        voList
    );
}
```

## 6. 前端配合说明

前端需要配合改动：

1. **首次加载**：不传 `cursorTime` 和 `cursorId`，获取最新数据
2. **滚动加载**：将上一页最后一条数据的 `createdAt` 作为 `cursorTime`，`id` 作为 `cursorId` 传入
3. **数据缓存**：使用 `react-query` 或 `swr` 管理无限滚动数据

---

## 7. 风险与注意事项

| 风险 | 应对措施 |
|------|----------|
| 数据删除导致游标失效 | 游标失效时，返回第一页数据 |
| 同时间戳多条数据 | 使用 `(created_at, id)` 组合游标解决 |
| 性能问题 | 确保索引 `idx_category_time` 存在 |
| 空结果处理 | 返回空数组和 `total=0` |
